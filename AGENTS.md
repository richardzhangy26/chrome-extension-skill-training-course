# Repository Guidelines

## Project Structure & Module Organization
- `chrome-extension/` holds the MV3 manifest source (`manifest.ts`), background service worker (`src/background`), and static assets (`public/`).
- `pages/` contains UI surfaces: `popup/`, `side-panel/`, `options/`, `new-tab/`, `devtools/`, `devtools-panel/`, plus injected UIs in `content-ui/` and `content-runtime/`.
- `packages/` contains shared libraries and tooling (e.g., `shared/`, `storage/`, `ui/`, `hmr/`, `dev-utils/`, `module-manager/`).
- `tests/e2e/` contains WebdriverIO test config and specs (`tests/e2e/specs/`).
- `dist/` is generated build output (load this folder in Chrome for testing).

## Architecture Overview

**Data Flow**: User visits a page with `trainTaskId` -> side panel extracts it -> background script authenticates via the `ai-poly` cookie -> API calls to `cloudapi.polymas.com` -> chat conversation with the AI assistant (text mode) or a WebSocket training-flow channel (voice mode).

**Key Components** (all under `pages/side-panel/src/` unless noted):

| Layer | File | Responsibility |
|-------|------|---------------|
| Background | `chrome-extension/src/background/index.ts` | Auth cookies, API proxy, `trainTaskId` extraction. Messages: `GET_AUTH`, `EXTRACT_TRAIN_TASK_ID`, `API_REQUEST` |
| Text Chat Hook | `hooks/useAgentChat.ts` | Conversation state machine: `IDLE -> FETCHING_STEPS -> RUNNING_CARD -> CHATTING -> COMPLETED` |
| Voice Chat Hook | `hooks/useVoiceAgentChat.ts` | Voice-mode state machine driving the WS + TTS pipeline; see `pages/side-panel/AGENTS.md` for the state list |
| LLM Service | `services/llm-service.ts` | AI dialogue generation, model management, student-answer generation, staged dialogue simulation |
| Bridge | `services/background-bridge.ts` | Side panel <-> background script communication |
| Main UI | `SidePanel.tsx` | Chat UI, mode toggle (text/voice), and modals |

See `pages/side-panel/AGENTS.md` for the full breakdown of the voice/TTS pipeline and LLM service internals.

**Storage Layer** (`packages/storage/lib/base/`): a Chrome storage wrapper with type safety and live sync.
```typescript
const storage = createStorage<DataType>('key', defaultValue, {
  storageEnum: StorageEnum.Local,
  liveUpdate: true,
});
// React: useSyncExternalStore pattern via getSnapshot() + subscribe()
```
All storage modules must be exported from `packages/storage/lib/impl/index.ts`. Key modules: `agent-session-storage` (trainTaskId/sessionId/stepId), `agent-chat-storage` (chat history), `llm-config-storage` (API credentials, model, student profiles, system prompt, dialogue-simulation content, knowledge base, voice/TTS settings), `agent-log-storage` (per-session conversation logs, `log_{timestamp}_{random}` ids).

**Dialogue Simulation & Knowledge Base** (`SimulationConfigModal.tsx`): accepts historical logs in `AI: / 用户:` format, or auto-generates them stage-by-stage via the LLM with a selectable student profile (`good`/`medium`/`bad`); a free-form knowledge base feeds the AI student role. Both are independently toggleable and persisted in `llmConfigStorage`.

**Polymas Integration**:
- Auth cookie: `ai-poly` from `hike-teaching-center.polymas.com`
- REST API base: `https://cloudapi.polymas.com`
- Voice training WebSocket: `wss://cloudapi.polymas.com/ai-tools/ws/v2/trainFlow`
- Required URL param: `trainTaskId` (e.g., `?trainTaskId=g2dlgQ4JOYimgZnNXxkb`)

## Framework Quirks & Important Conventions
- **DO NOT edit `manifest.json` directly.** Always edit `chrome-extension/manifest.ts`, as the JSON is auto-generated by the build pipeline.
- **Path Aliasing / Workspaces**: Always import shared code using the `@extension/*` namespace (e.g., `import { X } from '@extension/shared';` or `@extension/storage`). **Never** use relative paths to jump across workspace package boundaries (like `../../packages/shared/...`).
- **Feature Deletion**: Use `pnpm module-manager -d <feature-name>` (e.g., `pnpm module-manager -d popup`) to remove unused extension UI pages. This CLI safely archives the folder and cleans up the manifest automatically. Do not manually delete these folders.
- **Dependency Management**: To add a package globally, use `pnpm i <pkg> -w`. To add to a specific UI/package workspace, use `pnpm i <pkg> -F <workspace-name>` (e.g., `pnpm i lodash -F side-panel`).

## Build, Test, and Development Commands
- `pnpm dev`: development build with watch/HMR across workspaces.
- `pnpm build`: production build into `dist/`.
- `pnpm build:firefox`: Firefox build output.
- `pnpm zip`: build and package `dist` into a zip under `dist-zip/`.
- `pnpm clean`: Cleans `dist`, `node_modules`, and `.turbo` cache.
- `pnpm clean:install`: Performs a deep clean and fresh reinstall.
- `pnpm lint`: run ESLint across packages via Turborepo.
- `pnpm format`: run Prettier formatting.
- `pnpm type-check`: TypeScript type checks across workspaces.
- `pnpm e2e`: builds and runs end-to-end tests.

## Coding Style & Naming Conventions
- TypeScript + React (ESM); keep files consistent with existing naming.
- Indentation is 2 spaces; keep semicolons and trailing commas as in the codebase.
- React components use PascalCase; hooks use `useX` naming (e.g., `useAgentChat`).
- Linting/formatting: `eslint.config.ts` and Prettier via `pnpm lint` and `pnpm format`.

## ESLint Rules (Strict)

This project enforces strict ESLint rules. Pre-commit hooks will block commits with errors.

### Function Style (`func-style`)
- **Use arrow function expressions**, not function declarations
```typescript
// ❌ Wrong
function MyComponent() { ... }

// ✅ Correct
const MyComponent = () => { ... };
```

### Export Position (`import-x/exports-last`)
- **All exports must be at the end of the file**
```typescript
// ❌ Wrong - exports in the middle
export const foo = 1;
const bar = 2;

// ✅ Correct - exports at the end
const foo = 1;
const bar = 2;
export { foo };
```

### Accessibility (`jsx-a11y/*`)
- Clickable non-interactive elements need keyboard support:
```tsx
// ✅ Required attributes for clickable divs
<div
  onClick={handleClick}
  onKeyDown={e => e.key === 'Enter' && handleClick()}
  role="button"
  tabIndex={0}
  aria-label="Description"
/>
```
- Form labels must be associated with controls via `htmlFor` and `id`

### Unused Variables (`@typescript-eslint/no-unused-vars`)
```typescript
// ❌ Wrong
} catch (e) { console.log('error'); }

// ✅ Correct
} catch { console.log('error'); }
```

## Testing Guidelines
- E2E tests use WebdriverIO + Mocha (`tests/e2e/`).
- Place specs in `tests/e2e/specs/*.ts`.
- Run locally with `pnpm e2e` or `pnpm -F @extension/e2e e2e` for scoped runs.

## Commit & Pull Request Guidelines
- Follow Conventional Commits (example in history: `feat: ...`). Prefer short, descriptive subjects.
- PRs should include: a concise summary, test results/commands run, and screenshots for UI changes.
- Avoid committing `dist/` unless explicitly required for a release artifact.

## Environment Notes
- Node.js >= 22.15.1 and `pnpm@10.11.0` (see `package.json`).
- Load the extension for manual testing by selecting the `dist/` folder in `chrome://extensions`.
- **HMR freezes**: restart the dev server, or kill the `turbo` process if you see grpc errors.
- **Background script**: runs as a service worker — no DOM access, use `chrome.*` APIs only.
- **Environment variables**: edit `packages/env/.env.defaults`, register in `packages/env/src/index.ts`, import from `@extension/env`.
- **Tailwind**: extend via `packages/tailwind-config`; don't duplicate configs per workspace.
- **Chat UI styling**: gradient backgrounds, asymmetric rounded corners (`rounded-br-md`), animations — keep consistent with existing message bubbles.
