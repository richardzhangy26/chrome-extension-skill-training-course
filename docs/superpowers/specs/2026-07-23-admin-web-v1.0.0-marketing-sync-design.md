# Admin Web 官网内容同步至扩展 v1.0.0 设计

## 背景

Chrome 扩展已发布 v1.0.0（新增口语训练、能力训练 Pro、账号登录与云同步、网页控制台仪表盘等能力，详见根目录 `README.md` 的更新日志）。`admin_web` 官网首页与导航目前仍停留在 v0.8.0 文案（hero 徽标“v0.8.0·多角色并行训练”、下载按钮“下载 v0.8.0”），且：

- 导航/页脚仍展示未使用的“博客”入口。
- 没有反馈入口，用户无法直接图文反馈问题。
- 下载链接指向 GitHub Releases 列表页，而实际发布走 Gitee（`README.md` 已改为优先引导 Gitee v1.0.0 ZIP）。
- FAQ 中“数据会上传吗”的回答（“不会，均本地存储”）在账号登录云同步功能上线后已不准确。
- Chrome 应用商店版本审核较慢，目前仍是 v0.8.0，需要在官网提示用户优先用 ZIP。

## 目标

- 首页与导航文案同步到 v1.0.0：版本号、hero 文案、核心优势描述、统计数字、FAQ 内容。
- 关闭博客入口（配置开关，不删路由/数据）。
- 导航与页脚新增“问题反馈”入口，指向 Gitee Issues。
- 下载链接优先指向 Gitee v1.0.0 Release（ZIP），同时在首页与 FAQ 提示 Chrome 应用商店可用但审核中、版本落后。
- 修正 FAQ 中关于数据上传的过时表述。

## 非目标

- 不更换首页产品截图与功能轮播图（`features.tsx` 五张图、hero 大图）——这些资源托管在 Cloudflare R2 公开桶（`getR2AssetUrl`），本次会话没有上传权限，也不在没有授权的情况下改动共享基础设施。README 中新增的 `登录.png`/`控制台.png`/`能力训练pro展示.png`/`自定义模拟对话剧本.png` 暂不接入官网。
- 不新增或编造用户评价（testimonials）——现有 6 条是占位性质示例，不为凑合新功能编造虚假评价。
- 不改动 `features.tsx`/`features2.tsx` 现有文案——描述仍准确，没有过时问题。
- 不删除博客路由、组件或文章内容，只关闭入口。
- 不改动扩展仓库任何代码，只改 `admin_web`。
- 不涉及支付/定价方案改动。

## 方案选择

直接修改现有配置文件（`links.ts`/`website.ts`/`navbar-config.ts`/`footer-config.ts`）、locale 文案（`project.inlang/messages/{zh,en}.json`）和两处硬编码文案的组件（`faqs.tsx` 新增一条 FAQ、`stats.tsx` 改一个数字）。不引入新的配置 schema 字段，不新建组件，符合仓库“最小化改动范围”的约定。

被放弃的方案：新增独立的“下载中心”页面或重构首页版块顺序——当前诉求是内容同步而非信息架构调整，且截图资源受限，重构收益有限。

## 改动清单

### 配置（`admin_web/src/config/`)

- **`links.ts`**：
  - `DOWNLOAD_URL` 由 `${REPO_URL}/releases` 改为 `https://gitee.com/richardzhangy26/chrome-extension-skill-training-course/releases/tag/v1.0.0`（与 README 一致；后续发新版本需要同步手动更新这个常量，README 也是同样维护方式）。
  - 新增导出 `FEEDBACK_URL = 'https://gitee.com/richardzhangy26/chrome-extension-skill-training-course/issues'`。
  - 新增导出 `CHROME_WEB_STORE_URL = 'https://chromewebstore.google.com/detail/hike-teaching-center/mhhonaofiieikgebniaihgbphfplbhah?hl=zh-CN'`。
  - `REPO_URL` 保持 GitHub 地址不变（导航“GitHub”项、hero 徽标、次要按钮继续指向源码仓库）。

- **`website.ts`**：`blog.enable: true` → `false`。导航/页脚的博客入口由现有条件判断自动隐藏，博客路由与文章内容不动。

- **`navbar-config.ts`**：在 `GitHub` 之后新增一项 `{ title: m.nav_feedback(), href: FEEDBACK_URL, external: true }`。

- **`footer-config.ts`**：`resourcesItems` 在 `GitHub`、下载链接之后新增“问题反馈”“Chrome 应用商店”两项：
  ```ts
  const resourcesItems: MenuItemConfig[] = [
    { title: 'GitHub', href: REPO_URL, external: true },
    { title: m.home_hero_primary(), href: DOWNLOAD_URL, external: true },
    { title: m.nav_feedback(), href: FEEDBACK_URL, external: true },
    { title: m.nav_chrome_store(), href: CHROME_WEB_STORE_URL, external: true },
  ];
  ```
  博客条目因 `blog.enable=false` 自动不再出现，代码逻辑不用改。

### Hero 区新增 Chrome 应用商店提示（`admin_web/src/components/blocks/hero.tsx`)

在现有两个按钮（下载 / 查看使用指南）下方新增一行小字提示，整行作为链接指向 `CHROME_WEB_STORE_URL`：

```tsx
    target="_blank"
    rel="noopener noreferrer"

  </a>

```

## 验证与部署

1. 在 `admin_web/` 运行 `pnpm check`（Biome，只读检查）。
2. 在 `admin_web/` 运行 `pnpm build`，确认新增消息键编译无误、无 TypeScript 报错。
3. 本地 `pnpm dev` 走一遍首页：确认导航无“博客”、有“问题反馈”外链；hero 区版本号与 Chrome 商店提示展示正常；FAQ 新增第 6 条可展开；页脚资源分组含反馈与商店链接。
4. 不在本次任务中执行 `pnpm deploy`，除非用户明确要求上线。

## 风险与缓解

- **Gitee release 链接需要随版本手动更新**：与 README 现状一致，非本次新增维护成本，此处不做自动化。
- **Chrome 商店提示信息会过期**：一旦商店审核通过并更新到 v1.0.0，需要回来去掉这行提示或改文案；本设计不做版本联动判断（YAGNI，避免过度设计一个只有两个状态的开关）。
- **中英文案不同步**：改动清单已逐条列出 zh/en 对照，实现时按同一顺序成对编辑，避免漏改一侧。
